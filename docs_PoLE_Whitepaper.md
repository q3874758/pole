# 执行摘要

Proof of Live Engagement (PoLE) 是一个创新的 Layer 1 公链项目，旨在将 PC 游戏生态的真实玩家活跃度转化为可量化的链上价值。该项目通过玩家运行的分布式节点网络，采集游戏在线数据（以 Steam 公开 API 为核心），计算 Game Value Score (GVS)，并以原生代币 $POLE 形式奖励参与者。这一机制实现了“玩游戏即挖矿”的全新经济模型，覆盖所有 PC 客户端游戏，但优先奖励那些拥有公开、可验证实时活跃数据的游戏生态。

PoLE 的核心创新在于将玩家本身转变为区块链网络的验证节点，利用 Steam 当前在线玩家数作为主要信号源，通过分层 Tier 系统平衡数据可信度与全覆盖叙事。该项目结合了 DePIN（去中心化物理基础设施网络）、GameFi 和文化价值证明的多重优势，目标是构建一个去中心化的游戏经济飞轮。

从技术架构角度来看，PoLE 采用模块化设计，包括共识层（PoS + PoLE）、执行层（Rust 实现）和数据层（分布式节点采集、链上 oracle 聚合、GVS 计算、奖励发放）。经济模型方面，项目设计了可持续的代币通胀曲线、燃烧机制、难度调整系统和slash惩罚机制，以确保网络的长期健康发展。

本白皮书详细阐述了 PoLE 的技术架构、核心机制、经济模型、治理方案、安全考量和发展规划。项目预计在 2026 年至 2027 年上线主网，目标是吸引数万玩家节点参与网络验证，共同构建一个真正由玩家共建、共治、共享的游戏价值网络。

# 第一章 问题陈述与背景

## 1.1 PC 游戏生态现状

全球 PC 游戏市场正经历前所未有的繁荣时期。根据行业统计数据，2025 年全球 PC 游戏市场规模已突破 450 亿美元，玩家数量超过 10 亿人。这一庞大的生态系统涵盖了从 AAA 大作到独立小游戏的丰富多样性，形成了多元化、多层次的市场结构。Steam 作为全球最大的 PC 游戏数字发行平台，拥有超过 30,000 款游戏注册，每日在线玩家峰值超过 3600 万人，构成了 PC 游戏生态的核心枢纽。

然而，尽管 PC 游戏生态创造了巨大的经济价值和社会文化影响，普通玩家的真实参与却从未被转化为可流通的经济价值。玩家们投入的时间、精力和热情构成了游戏生态活力的根基，但他们既无法从自己的参与中获得直接回报，也无法将这种参与价值进行量化或交易。这种价值分配的失衡导致了生态效率的低下，也阻碍了更多潜在参与者加入游戏社区。

## 1.2 现有解决方案的局限性

当前区块链与游戏领域的结合主要呈现两种模式，但都存在明显的局限性。第一种是以 Axie Infinity、StepN 为代表的链游模式，将游戏玩法与代币经济深度绑定，但这类游戏往往依赖于中心化的游戏服务器，数据可操纵性强，且游戏体验与传统游戏存在较大差距。第二种是以游戏资产 NFT 化为核心的 Web3 模式，将游戏道具、角色进行链上确权，但这种方式仅关注资产所有权，并未触及玩家参与本身的价值实现。

传统公链项目在设计游戏相关功能时，往往倾向于构建全新的区块链游戏，或者将现有游戏进行 NFT 化改造，却忽略了一个根本性问题：这些方法都需要游戏开发商的主动参与和适配，门槛较高，且难以覆盖海量的已有 PC 游戏生态。更重要的是，无论采用哪种方式，都存在一个共同的缺陷——它们依赖的中心化数据源（如游戏官网 API、第三方统计平台）存在被操纵的风险，数据的真实性和可信度难以保证。

## 1.3 玩家激励缺失的深层问题

游戏生态的活力来源是真实玩家的积极参与

## 1.1.1 DePIN 赛道格局

当前 DePIN 赛道呈现出快速发展的态势，涵盖分布式无线网络（Helium）、去中心化地图（Hivemapper）、算力网络（Render Network）等多个细分领域。PoLE 作为游戏领域的 DePIN 基础设施，与这些项目具有相似的激励结构——通过代币激励机制协调分布式资源贡献者，构建传统中心化方案无法提供的基础设施服务。

**与现有方案的差异化**：与需要游戏开发商主动集成区块链的传统 GameFi 方案不同，PoLE 采用被动数据采集模式，通过多个游戏平台的公开 API 获取公开数据，无需游戏开发商适配即可提供服务。目前 PoLE 主要支持的数据源包括 Steam（全球最大 PC 游戏平台）、Epic Games Store、EA app、GOG.com 等平台的公开数据。这种"无许可"特性大大降低了生态扩展的门槛，使 PoLE 能够快速覆盖海量已有 PC 游戏生态。

## 1.2 现有解决方案的局限性，但这种参与长期以来处于“无偿贡献”状态。玩家的在线时间、游玩频次、社区互动等行为创造了游戏的热度和价值，却无法获得任何直接经济回报。这种激励机制缺失导致了多重负面效应：核心玩家的高投入行为无法得到合理认可，轻度玩家的参与动力不足，游戏社区的活跃度难以持续提升。

与此同时，游戏数据的高度中心化进一步加剧了这个问题。游戏热度指标、玩家数量统计等关键数据完全由游戏平台和开发商掌控，普通用户无法获取原始数据，更无法验证数据的真实性。这种信息不对称不仅影响了玩家对游戏的判断，也为市场操纵和数据欺诈留下了空间。

## 1.4 PoLE 的解决方案

针对上述问题，PoLE 提出了一套全新的解决思路。项目的核心目标是建立一个去中心化的数据采集和价值评估网络，将玩家的真实游戏参与转化为可量化的链上价值。具体而言，PoLE 通过以下方式实现这一目标：

首先，PoLE 将游戏玩家转变为区块链网络的验证节点。任何安装了 PoLE 客户端软件的玩家都可以成为网络节点，通过运行节点软件来采集和验证游戏数据。这种设计不仅实现了真正的去中心化，还大大降低了参与门槛，让每个玩家都有可能成为网络的建设者。

其次，PoLE 采用多数据源交叉验证机制，确保数据的真实性和可信度。项目优先采用 Steam 公开 API 作为主要数据来源，利用其提供的实时在线玩家数、峰值并发数等公开数据进行 GVS 计算。同时，节点网络通过中位数共识机制对数据进行聚合和验证，有效防止单个节点的数据操纵。

最后，PoLE 设计了一套可持续的经济模型，将数据采集和验证工作转化为可获得代币奖励的经济活动。玩家通过运行节点贡献计算资源和网络带宽，采集和验证游戏数据，即可获得 $POLE 代币奖励。这种“玩游戏即挖矿”的模式真正实现了玩家参与价值的即时变现。

# 第二章 技术架构

## 2.1 整体架构设计

PoLE 采用模块化的 Layer 1 公链架构，将网络功能划分为多个独立且可插拔的模块。这种设计不仅提高了系统的可维护性和可升级性，还允许不同模块独立演进，优化整体性能。整体架构分为四个核心层次：数据采集层、网络共识层、执行环境和应用层。

数据采集层是 PoLE 区别于传统公链的核心创新所在。该层由分布式节点网络组成，每个节点通过运行专门的采集软件来获取游戏平台的公开数据。数据采集层的设计重点在于确保数据的真实性、完整性和时效性，通过节点间的交叉验证来排除异常数据和欺诈行为。

网络共识层采用 PoS（Proof of Stake）与 PoLE（Proof of Live Engagement）相结合的混合共识机制。PoS 负责区块生产的确定性和网络整体的安全性，而 PoLE 则专门处理游戏数据采集和验证的共识。这种分工设计使得共识机制能够更加专注于各自的任务，提高整体效率。

执行环境基于 Rust 编程语言构建，提供了高性能、低延迟的交易处理能力。Rust 的内存安全特性和零成本抽象使得 PoLE 能够实现企业级的代码质量，同时保持优异的性能表现。

应用层为开发者和用户提供了丰富的接口和工具，包括 SDK、RPC 接口、浏览器插件等。通过这些工具，开发者可以轻松构建基于 PoLE 的去中心化应用，用户也可以方便地参与网络治理和使用网络服务。

## 2.2 数据采集与聚合机制

PoLE 的数据采集机制是整个系统的价值基础。该机制的核心思想是利用分布式节点网络来采集和验证游戏平台的公开数据，而非依赖单一的中心化数据源。每个加入网络的节点都会运行数据采集程序，持续监控目标游戏平台的公开 API 接口，获取实时的玩家活跃数据。

数据采集过程遵循严格的规范和协议。首先，节点软件会定期向目标 API 发送请求，获取最新的玩家在线数量、服务器状态等信息。然后，节点会对获取的数据进行初步处理，包括数据格式标准化、时间戳添加等操作。最后，处理后的数据会被打包并广播到网络中的其他节点。

为了确保数据的真实性和防止作弊，PoLE 采用了多节点交叉验证机制。当一个节点上报数据时，网络会随机选取一组其他节点对同一数据源进行独立采集。如果多个节点上报的数据高度一致（偏差在可接受范围内），则该数据被认为是可信的并被接受；如果存在显著差异，则会触发异常检测流程，对问题节点进行标记和惩罚。

数据聚合是采集流程的最终环节。网络中的验证节点会对来自多个采集节点的数据进行处理，采用中位数算法剔除极端值，计算出最终的游戏价值得分（Game Value Score）。这种设计确保了即使部分节点被攻击或出现故障，整体数据仍然保持可靠性。

## 2.3 共识机制详解

PoLE 的共识机制是整个网络的核心支柱，它结合了 PoS（权益证明）和 PoLE（活跃证明）两种机制的优点，形成了独特的三层共识结构。

第一层是区块生产共识，采用 PoS 机制。区块生产者（Validator）需要质押一定数量的 $POLE 代币作为保证金，有资格参与区块生产。区块生产的权重与质押金额成正比，但存在最大质押上限，防止单一节点过度集中。这种设计既保证了网络的安全性，又实现了去中心化。

第二层是数据验证共识，采用 PoLE 机制。该层专门负责对采集的游戏数据进行验证和确认。验证过程采用拜占庭容错（BFT）风格的多数共识，需要超过三分之二的节点对数据达成一致才能确认。这种设计确保了即使有最多三分之一的节点被攻击或行为不端，网络仍然能够正确运行。

第三层是争议解决共识，用于处理数据验证过程中出现的争议。当节点对某一数据点存在分歧时，网络会启动争议解决流程，随机选取更多节点进行独立验证，直到达成明确共识。这种机制为网络提供了额外的安全缓冲。

## 2.4 执行层技术实现

执行层采用 Rust 编程语言开发，充分利用了 Rust 的性能优势和安全特性。Rust 的所有权系统和生命周期检查能够在编译时消除大部分常见的编程错误，如空指针引用、内存泄漏等，大大提高了代码的可靠性。同时，Rust 接近汇编的执行效率使得 PoLE 能够实现每秒处理数千笔交易的目标。

交易执行采用 Actor 模型进行并发处理。Actor 模型是一种经典的并发编程范式，每个 Actor 拥有独立的执行状态，通过消息传递与其他 Actor 通信。这种设计天然适合分布式系统的开发，能够有效避免锁竞争和状态不一致的问题。

状态管理采用账户模型，每个账户拥有独立的余额和存储空间。智能合约以 WASM（WebAssembly）形式部署，用户可以自定义合约逻辑。这种设计既保证了灵活性，又通过预编译合约优化了常见操作的执行效率。

网络通信采用 libp2p 库，实现了去中心化的节点发现和连接管理。节点之间通过加密通道进行通信，支持多种传输协议，包括 TCP、QUIC 等。这种设计确保了网络在各种网络环境下的可用性和抗审查性。

## 2.5 模块化架构设计

PoLE 采用高度模块化的 Layer 1 公链架构，将网络功能划分为四个独立可插拔的层。这种设计不仅提高了系统的可维护性和可升级性，还允许不同模块独立演进，优化整体性能。

### 2.5.1 数据采集层（Sensing Layer）

数据采集层是 PoLE 网络的数据入口，负责从多种数据源采集游戏活跃度数据。该层采用插件化设计，支持灵活扩展新的数据源。

**平台 API 集成模块**：通过 Steam Web API 获取游戏的实时在线玩家数、峰值玩家数、历史趋势等数据；Epic Games Store 集成通过页面爬取和数据聚合服务获取游戏活跃度数据；第三方统计平台集成与 PlayerIGN、SteamDB 等数据平台合作；社区验证输入为没有公开 API 的游戏提供社区验证机制。

**数据格式标准化模块**：所有采集到的原始数据经过标准化处理后，进入统一的内部格式，确保跨平台数据的一致性和可操作性。

**轻节点采集**：PoLE 网络支持轻节点进行数据采集。轻节点运行在普通玩家电脑上，CPU 占用控制在 5% 以下，内存使用不超过 500MB，不会影响用户的正常游戏体验。轻节点采集的数据经过验证后与其他节点数据聚合，确保网络数据的完整性。

### 2.5.2 数据可用性层（Data Availability Layer）

数据可用性层负责存储和管理采集到的原始数据，采用分布式存储架构确保数据的持久性和抗审查性。

**分布式数据存储**：采用纠删码（Erasure Coding）技术，将数据分散存储在多个节点上，数据冗余度可配置，默认设置为 4+2 模式（4 个数据分片 + 2 个校验分片），任意 4 个分片即可恢复完整数据。

**自愿存储机制**：PoLE 网络鼓励节点自愿贡献闲置存储空间。任何运行轻节点或存储节点的玩家都可以开启存储功能，利用闲置磁盘为网络提供数据保存服务。存储贡献者根据存储空间大小和数据可用性获得奖励。

**数据索引服务**：构建高效的数据索引，支持按时间、游戏、平台等多维度查询，实时更新索引确保新数据可立即被查询。

**历史数据归档**：超过 90 天的数据进行归档处理，归档数据存储在冷存储中降低存储成本。

### 2.5.3 计算验证层（Execution Layer）

计算验证层是 PoLE 网络的核心处理引擎，负责数据处理、验证和 GVS 计算。

**AI 驱动的异常检测引擎**：采用轻量级机器学习模型进行实时异常检测，模型运行在每个验证节点上独立进行推理，检测结果通过零知识证明验证确保计算正确性。

**GVS 计算核心模块**：实现标准化的 GVS 计算公式，支持可配置的参数调整，通过治理投票修改权重，计算结果可审计、可追溯。

**多节点交叉验证模块**：当单个节点的数据被质疑时，启动多节点交叉验证，采用 BFT 共识的子集进行快速验证。

### 2.5.4 结算层（Settlement Layer）

结算层负责链上激励结算和状态确认，是 PoLE 网络与区块链交互的接口。

**区块生产和共识模块**：基于 Cosmos SDK 的 Tendermint 共识，实现 PoLE+DPoS 混合共识机制，支持快速 finality，区块确认时间小于 3 秒。

**代币分发和燃烧模块**：自动执行通胀产生的区块奖励分发，实现交易手续费的燃烧机制，支持惩罚性 slash 的代币处理。

**治理投票模块**：支持链上提案和投票，实现代币加权的投票机制，自动执行通过的提案。

### 2.5.5 模块间通信协议

各层之间通过标准化的通信协议进行交互，确保系统的松耦合性。

**内部消息队列**：采用 gRPC 进行层间通信，消息格式采用 Protocol Buffers 定义，支持同步和异步两种调用模式。

**事件驱动架构**：各层之间通过事件总线进行解耦，支持事件的持久化和重放，便于系统调试和审计。

**服务发现机制**：动态服务注册和发现，支持服务的水平扩展，内置负载均衡和故障转移。

## 2.6 跨链与兼容性设计

PoLE 网络积极整合人工智能技术，以提升数据处理效率和网络安全性。AI 模块作为可选增强功能，运行在计算验证层，为整个网络提供智能化的支持。

### 2.6.1 异常检测与反作弊

AI 驱动的异常检测引擎能够识别节点上报数据中的异常模式。系统采用多层检测机制，从数据完整性、行为模式、时间序列等多个维度进行综合分析。

**第一层：数据完整性检测**

- 验证 Steam API 返回的数据格式是否符合平台规范
- 检查数据时间戳的合理性（不允许未来时间或超过 24 小时的历史数据）
- 交叉验证多个数据源的聚合数据是否一致
- 检测数据字段的完整性，确保无缺失值或异常值

**第二层：行为模式分析**

- 机器学习模型分析节点历史行为，识别可疑的数据操纵模式
- 建立节点行为画像，检测瞬间玩家数突变、规律性刷分等异常行为
- 监控节点的活跃时间分布，识别非人类操作特征
- 分析同一硬件指纹下的多开行为，检测 Sybil 攻击

**第三层：网络协同检测**

- 多节点数据交叉验证，识别串通作弊的节点集群
- 地理位置异常检测，识别使用 VPN 或代理的隐藏身份
- 网络拓扑分析，发现试图操纵共识的恶意节点网络

### 2.6.2 GVS 智能计算

传统的 GVS 计算基于固定公式和静态参数，但在快速变化的游戏市场中，这种方法可能无法准确反映游戏的真实价值。引入 AI 模型后，系统能够综合分析更多元的指标，并动态适应市场变化。

**多维度价值评估**

AI 驱动的 GVS 计算引入以下额外评估维度：玩家留存率分析（1 日、7 日、30 日留存率）、付费转化率评估、社区活跃度分析（Discord、Reddit、Twitter 等平台）、市场趋势预测、生态互补性评估。

**动态权重调整**

AI 模型能够根据市场环境自动调整各指标的权重。当整体市场下行时，增加「付费转化率」权重；当某类游戏集中上线时，降低「稀缺性」权重；当网络规模扩大时，增加「去中心化贡献」权重。

### 2.6.3 技术实现规格

AI 模块采用轻量级设计，确保不影响网络性能。模型推理延迟目标为小于 10 毫秒，CPU 占用控制在 5% 以下，内存占用低于 500MB，模型大小控制在 100MB 以内，更新周期为每周一次。AI 推理主要在本地节点执行，模型以加密形式分发确保算法隐私，输入数据本地处理仅提交推理结果和零知识证明验证计算正确性。

# 第三章 核心机制

## 3.1 Game Value Score (GVS) 体系

Game Value Score（GVS）是 PoLE 网络量化游戏参与价值的核心指标。GVS 的设计目标是准确反映一款游戏的真实活跃度和社区价值，同时防止数据操纵和作弊行为。GVS 的计算综合考虑了多个维度的因素，包括实时在线人数、历史活跃趋势、社区参与度等。

GVS 的基础计算公式如下：

**GVS = Base_GLV × Tier_Coefficient × Time_Decay × Coverage_Bonus**

其中，Base_GLV（Game List Value）代表游戏的基准价值，根据游戏的当前在线玩家数、峰值玩家数、历史平均玩家数等指标计算得出。Tier_Coefficient 是游戏分层系数，根据游戏的数据可信度和社区规模确定。Time_Decay 是时间衰减因子，用于调整长期表现与近期表现之间的权重。Coverage_Bonus 是节点覆盖加成，当某一游戏有更多节点进行数据采集时，该游戏的 GVS 会获得额外加成。

GVS 的计算采用滚动窗口机制，通常以 24 小时为周期进行重算。这种设计既能反映游戏的即时热度，又能平滑短期波动带来的噪音。同时，GVS 还会与历史数据进行对比，识别异常模式并进行调整。

## 3.2 Tier 分层系统

为了平衡数据可信度与全覆盖叙事，PoLE 采用了三层 Tier 分层系统。该系统根据游戏数据的可获取性和可信度，将游戏划分为不同的层级，每个层级对应不同的验证权重和奖励系数。

**Tier 1：核心验证层**

Tier 1 包括拥有可靠公开 API 数据的大型游戏平台，其中 Steam 是最重要的数据来源。被归入 Tier 1 的游戏需要满足以下条件：拥有公开、可验证的实时在线玩家数据；数据更新频率不低于每 5 分钟一次；历史数据可追溯且无明显操纵痕迹。Tier 1 游戏的 GVS 计算权重为 1.0，代表最高级别的数据可信度。

典型的 Tier 1 游戏包括 Steam 平台上当前在线人数超过 1000 人的所有游戏。这类游戏拥有活跃的玩家社区和稳定的流量，是 PoLE 网络的主要价值支撑。

**Tier 2：增强验证层**

Tier 2 包括数据可得但可靠性略低的游戏。这些游戏可能缺乏高频实时数据，或者数据来源相对单一，但仍然能够提供有价值的活跃度信号。Tier 2 的权重范围为 0.3 至 0.6，具体权重根据数据质量动态调整。

Tier 2 的典型代表包括：提供每日或每周玩家统计的游戏；拥有多个独立数据来源但缺乏统一 API 的游戏；新兴的热门游戏但数据历史较短。

**Tier 2 数据来源与计算方式**：
- 数据来源：第三方统计平台（如 SteamCharts、PlayerCount）、游戏官网定期公布的运营数据、多平台数据交叉对比
- 计算方式：Base_GLV 按日/周平均玩家数计算，乘以 0.3-0.6 的可信度系数
- 验证要求：至少 3 个独立节点上报数据，偏差不超过 30%

**Tier 3：社区验证层**

Tier 3 涵盖无法获取公开实时数据的游戏。对于这类游戏，GVS 的计算主要依赖于节点网络的社区验证机制。节点运营者可以通过手动上报、第三方数据整合等方式提供游戏活跃度信息，但需要经过更严格的验证流程。Tier 3 的权重范围为 0.05 至 0.15。

Tier 3 的设置确保了 PoLE 网络的包容性，即使是最小众的游戏也能参与到这个价值网络中。但同时，较低的权重也反映了这类数据相对较低的置信度。

**Tier 3 数据来源与计算方式**：
- 数据来源：节点手动上报（需提供截图/录屏等证据）、社区投票确认、第三方数据分析报告
- 计算方式：Base_GLV 按社区评估的活跃度等级（1-5星）计算，乘以 0.05-0.15 的低可信度系数
- 验证要求：至少 10 个独立节点上报数据，且超过 70% 的节点数据一致，需经过争议解决流程

## 3.3 节点运营机制

PoLE 网络采用多层次节点架构，支持灵活参与。节点分为四种类型：验证节点（Validator）、轻节点（Light Node）、存储节点（Storage Node）和采集节点（Collector）。这种分层设计让普通玩家可以用普通电脑参与网络，同时保持网络的安全性。

### 验证节点

验证节点负责区块生产、数据最终确认和网络治理，需要较高的硬件配置和稳定的网络连接。

**硬件要求**：CPU 8 核以上、内存 16GB 以上、存储 500GB SSD 以上、网络带宽 100Mbps 以上。

**质押要求**：初始最低要求 10,000 $POLE，确保节点有足够的经济激励遵守网络规则。

**在线要求**：99% 以上的在线时间，离线会导致质押扣除。

### 轻节点（Light Node）

轻节点是 PoLE 网络的创新设计，允许普通玩家在不影响日常使用的情况下参与网络。轻节点无需质押，可以灵活开启以下功能：

**数据采集**：采集游戏平台公开 API 数据，门槛极低，普通游戏玩家的个人电脑即可胜任。CPU 占用控制在 5% 以下，内存使用不超过 500MB，完全不影响正常游戏体验。

**自愿存储**：利用闲置磁盘空间为网络提供分布式存储。采用纠删码（Erasure Coding）技术，将数据分散存储在多个节点上，即使部分节点离线也能恢复完整数据。存储配额可配置，默认 10GB。

**轻量验证**：可选功能，质押 100 $POLE 即可参与数据验证。验证权重与质押量成正比，但计算量远小于全验证节点。

### 存储节点

存储节点专门为网络提供数据存储服务，不需要运行完整的区块链客户端。任何人可以自愿贡献闲置存储空间，获得存储奖励。

**存储机制**：采用 4+2 纠删码模式，4 个数据分片 + 2 个校验分片，任意 4 个分片即可恢复完整数据。

**存储配额**：默认 10GB，可根据个人磁盘空间自行调整。

**数据保留**：默认 30 天自动清理，减少存储压力。

### 采集节点

采集节点专注于游戏数据采集，适合资源有限但希望参与网络的轻度用户。

**硬件要求**：最低配置即可，一台能够运行 PoLE 客户端的电脑。

**无需质押**：采集节点不需要质押任何代币，降低参与门槛。

### 节点类型对比

| 节点类型 | 质押要求 | 硬件要求 | 奖励比例 | 主要职责 |
|---------|---------|---------|---------|---------|
| 验证节点 | 10,000+ | 8核/16GB/500GB SSD | 100% | 区块生产、数据确认、治理 |
| 轻节点 | 可选 100+ | 普通PC (CPU<5%) | 30-60% | 采集/存储/验证 |
| 存储节点 | 0 | 闲置磁盘 10GB+ | 10% | 数据分散存储 |
| 采集节点 | 0 | 最低配置 | 15-20% | 数据采集上报 |

### 节点激励

节点运营的激励来源于三部分：数据贡献奖励、区块生产奖励和存储服务奖励。

**数据贡献奖励**：根据节点采集数据的质量、数量和时效性计算。不同节点类型获得不同比例的奖励：

| 节点类型 | 数据采集奖励 | 存储奖励 | 验证奖励 |
|---------|------------|---------|---------|
| 验证节点 | 60% | - | 100% |
| 轻节点 | 30-60% | 10% | 30-60% |
| 存储节点 | - | 10% | - |
| 采集节点 | 15-20% | - | - |

**Tier 加成**：Tier 1 游戏数据权重最高（1.0x），Tier 2 为 0.3-0.6x，Tier 3 为 0.05-0.15x。

**区块生产奖励**：由验证节点共享，按照质押比例分配，设置单节点上限防止过度集中。

**存储服务奖励**：按贡献的存储空间和数据可用性计算，确保网络数据的持久性。

**质量系数**：节点历史表现（准确率、在线时间）会影响最终获得的奖励金额。

## 3.4 难度调整机制

PoLE 借鉴了比特币的难度调整机制，设计了一套自适应网络奖励系统。该机制的核心目标是保持网络整体收益率（APR）的稳定，确保参与者的投入产出比维持在合理范围内。

难度调整的基本原理如下：当网络总 GVS（TNGV）高于目标值时，单位数据贡献能够获得的奖励会相应降低；当 TNGV 低于目标值时，奖励会相应提高。这种设计形成了一个负反馈调节机制，使得网络能够自动适应参与者数量的变化。

具体的难度调整公式为：

**Current_Reward = Base_Reward × (Target_TNGV / Current_TNGV)^0.5**

公式中的指数 0.5 是调整系数，用于平滑奖励变化的幅度，避免出现过度的波动。目标 APR 设定为 5% 至 15% 之间，这是一个兼顾激励效果和网络可持续性的合理区间。

难度调整的周期设定为每个 Epoch（约 1 周），给参与者足够的时间来调整策略。同时，调整幅度设置了上限，单次调整的变化不超过 20%，防止出现剧烈的收益波动。

## 3.5 惩罚与 Slash 机制

为了维护网络的诚信和数据的可靠性，PoLE 设计了严格的惩罚机制。任何节点的不当行为都会受到相应的处罚，严重者甚至会被没收全部质押。

**偏差处罚**

当节点上报的数据与网络共识值存在显著偏差时，会触发偏差处罚。具体的阈值设定为：偏差超过 20% 但低于 50% 时，扣除节点 5% 的质押；偏差超过 50% 时，扣除节点 20% 的质押。这种渐进式的处罚设计既给予节点一定的容错空间，又对严重作弊行为形成有力震慑。

**连续违规处罚**

对于连续违规的节点，系统会采取更严厉的处罚措施。连续 3 次触发偏差处罚的节点将被禁赛 7 天，期间无法参与数据采集和验证。连续 5 次违规的节点将被永久移除网络，其全部质押将进入国库。

**诽谤处罚**

为了防止节点通过诽谤其他节点来获取不当利益，PoLE 还设计了诽谤处罚机制。如果节点被证实存在恶意诽谤其他节点的行为，将被扣除 10% 的质押作为处罚。

**轻节点特殊规则**

轻节点和采集节点无需质押，但仍然会受到声誉惩罚：

- 数据异常：标记为低信誉节点，降低奖励分配
- 长期离线：移除活跃节点列表
- 作弊行为：加入黑名单，禁止参与网络

# 第四章 代币经济模型

## 4.1 代币概述

$POLE 是 PoLE 网络的原生功能性代币，承担着网络运行的价值媒介和激励载体双重角色。作为功能性代币，$POLE 的设计重点在于支撑网络的核心功能，而非投机炒作。

**代币基本参数**

$POLE 代币的完整参数如下：代币名称为 POLE（Proof of Live Engagement）；代币符号为 $POLE；总供应量为 10 亿枚；代币标准为 SRC-20（PoLE 原生代币标准，兼容 ERC-20）；小数精度为 18 位。

**代币分配**

$POLE 代币的初始分配遵循以下比例：节点奖励池占比 60%，共 6 亿枚；生态发展基金占比 20%，共 2 亿枚；社区激励计划占比 15%，共 1.5 亿枚；团队与早期投资者占比 5%，共 0.5 亿枚。

这种分配比例的设计体现了 PoLE 对社区参与的重视。60% 的代币直接分配给节点运营者，确保了网络参与者能够获得足够的经济激励。20% 的生态基金用于支持项目生态的发展，包括开发者资助、DApp 激励、市场推广等。15% 的社区激励用于空投、用户获取和社区建设。5% 的团队分配确保了核心贡献者有足够的动力长期投入项目建设。

**代币解锁时间表**

为确保代币分配的透明性和长期可持续性，PoLE 设计了如下解锁时间表：

| 参与方 | 分配比例 | 锁定期 | 释放周期 | 备注 |
|--------|----------|--------|----------|------|
| 节点奖励池 | 60% | 无锁定期 | 按通胀曲线逐步释放 | 初始流通量 0 |
| 生态发展基金 | 20% | 无锁定期 | 按季度释放 | 用于生态建设 |
| 社区激励计划 | 15% | 无锁定期 | 按活动释放 | 空投及激励 |
| 团队 | 3% | 12个月 | 24个月线性释放 | 核心开发团队 |
| 早期投资者 | 2% | 6个月 | 12个月线性释放 | 天使及种子轮 |

**解锁细节说明**：

- **团队代币（3%）**：初始锁定12个月，解锁后按月线性释放24个月。团队代币的长期锁定体现了核心贡献者对项目长期发展的信心。

- **早期投资者代币（2%）**：初始锁定6个月，解锁后按月线性释放12个月。较短的投资人锁定期有助于在主网上线初期建立健康的流通量。

- **初始流通量预估**：主网上线时，生态发展基金和社区激励计划中预计有约 5% 的代币作为初始流动性提供，其余95%处于锁定期。这一设计确保了市场不会出现过度抛压。

## 4.2 通胀曲线设计

PoLE 的代币通胀设计是整个经济模型的核心环节。合理的通胀机制既要保证网络启动阶段有足够的代币流通，又要防止长期过度通胀稀释持有者价值。

**通胀曲线公式**

PoLE 采用指数衰减型通胀曲线，通胀率随时间推移逐步降低。具体的通胀公式为：

**Annual_Inflation_Rate = Initial_Inflation × (1/2)^(Year / 2)**

其中，Initial_Inflation 为初始年通胀率，设定为 20%。按照公式计算：第一年的通胀率为 20%，第二年同样为 20%；第三至第四年降至 10%；第五至第六年降至 5%；第七至第八年降至 2.5%；第九至第十年降至 1.25%；第十一年及以后降至约 1% 以下。

**技术实现**

通胀模块（x/mint）负责计算和分配新铸造的代币。核心计算逻辑采用以下公式实现：

```go
func CalculateInflation(year int, initialRate sdk.Dec, decayFactor sdk.Dec) sdk.Dec {
    exponent := sdk.NewDec(int64(year)).Quo(sdk.NewDec(2))
    decay := decayFactor.Power(uint64(exponent.RoundInt()))
    return initialRate.Mul(decay)
}
```

其中，decayFactor 设定为 0.5，表示每两年通胀率减半。模块还实现了分布参数管理，包括验证节点奖励、委托奖励、社区池资金等各项分配比例。

**通胀策略的合理性分析**

20% 的初始年通胀率看似较高，但实际上这是一个经过深思熟虑的设计。首先，高通胀率在网络启动阶段是必要的，因为此时节点数量较少，需要通过较高的单位激励来吸引早期参与者。其次，早期参与者承担了更高的风险（网络可能失败）和更大的投入（需要购买硬件、投入时间），理应获得更高的回报。最后，随着网络成熟和规模扩大，通胀率会自动下降，这是一个平滑的过渡过程。

## 4.3 代币燃烧机制

为了对抗通胀压力并创造代币价值捕获机制，PoLE 设计了多层次的代币燃烧体系。燃烧机制的核心思想是将部分流通中的代币永久移除，从而降低净通胀率，甚至在某些条件下实现通缩。

**技术实现**

燃烧模块（x/burn）实现了完整的代币燃烧逻辑，包括三种燃烧类型：

```go
// 计算交易燃烧
func CalculateFeeBurn(feeAmount sdk.Int, burnPercentage sdk.Dec) sdk.Int {
    return feeAmount.Mul(burnPercentage.Quo(sdk.NewDec(100)))
}

// 计算奖励燃烧
func CalculateRewardBurn(rewardAmount sdk.Int, threshold int64, burnPercentage sdk.Dec) sdk.Int {
    if rewardAmount.GT(sdk.NewInt(threshold)) {
        excess := rewardAmount.Sub(sdk.NewInt(threshold))
        return excess.Mul(burnPercentage.Quo(sdk.NewDec(100)))
    }
    return sdk.ZeroInt()
}

// 计算治理燃烧
func CalculateGovernanceBurn(lockedAmount sdk.Int, burnPercentage sdk.Dec) sdk.Int {
    return lockedAmount.Mul(burnPercentage.Quo(sdk.NewDec(100)))
}
```

**交易燃烧**

每笔在 PoLE 网络上发生的交易都需要支付手续费，而手续费的 20% 至 30% 将被永久燃烧。具体的燃烧比例由治理投票决定，初期设定为 25%。这意味着如果有用户支付了 1 $POLE 的手续费，其中 0.25 $POLE 将被燃烧，0.75 $POTE 进入奖励池。

交易燃烧机制创造了一个正向的价值捕获循环：网络使用越活跃，燃烧的代币越多，剩余代币的价值就越高。这种机制与以太坊的 EIP-1559 燃烧模型类似，已经在实践中被证明能够有效提升代币价值。

**奖励燃烧**

除了交易燃烧外，PoLE 还设计了额外的奖励燃烧机制。当节点获得的数据贡献奖励超过一定阈值时，超出部分将按一定比例燃烧。例如，如果某节点在一个 Epoch 内获得的奖励超过 10,000 $POLE，则超出部分的 10% 将被燃烧。

奖励燃烧机制的作用是双重的：一方面，它防止了早期巨鲸通过大量运营节点来快速获取代币；另一方面，它创造了一个更加公平的收入分配，确保小额参与者也能获得合理的回报。

**治理燃烧**

第三个燃烧渠道是治理参与燃烧。PoLE 鼓励代币持有者积极参与网络治理，每进行一次治理投票，参与者需要锁定一定数量的代币。这些被锁定的代币在投票期间无法流通，相当于暂时退出流通。当治理提案获得通过后，被锁定的代币中将有一小部分（例如 1%）被永久燃烧，作为参与治理的“费用”。

**燃烧参数**

| 参数 | 默认值 | 可调范围 | 说明 |
|------|--------|----------|------|
| 交易燃烧比例 | 25% | 20%-30% | 手续费中燃烧的比例 |
| 奖励燃烧阈值 | 10,000 $POLE | 5,000-50,000 | 触发奖励燃烧的阈值 |
| 奖励燃烧比例 | 10% | 5%-20% | 超出阈值部分的燃烧比例 |
| 治理燃烧比例 | 1% | 0.5%-2% | 治理锁定代币的燃烧比例 |

**燃烧效果模拟**

为了评估燃烧机制的实际效果，我们进行了以下模拟分析。假设网络在第一年达到 10,000 个活跃节点，每天平均处理 50,000 笔交易，平均每笔交易的手续费为 0.1 $POLE：

年交易手续费收入 = 50,000 × 365 × 0.1 = 1,825,000 $POLE
年燃烧量 = 1,825,000 × 25% = 456,250 $POLE
年铸造量 = 200,000,000 $POLE
净通胀量 = 200,000,000 - 456,250 = 199,543,750 $POLE
实际通胀率 = 19.95%

可以看出，在网络初期阶段，交易燃烧只能略微降低通胀率。但随着网络规模扩大和使用频率增加，燃烧效果会越来越显著。假设在第五年，网络达到 100,000 个节点，每天处理 500,000 笔交易：

年交易手续费收入 = 500,000 × 365 × 0.1 = 18,250,000 $POLE
年燃烧量 = 18,250,000 × 25% = 4,562,500 $POLE
年铸造量 = 50,000,000 $POLE
净通胀量 = 50,000,000 - 4,562,500 = 45,437,500 $POLE
实际通胀率 = 6.57%

到了第五年，实际通胀率已经降至 6.57%，显著低于名义通胀率 5%。这意味着即使在通胀率下降的情况下，燃烧机制也能进一步增强代币的稀缺性。

## 4.4 奖励分配模型

PoLE 的奖励分配模型是整个经济系统的发动机，它决定了参与者如何获得回报，以及回报的分配比例是否公平合理。

**奖励池来源**

PoLE 网络的奖励池资金来源于两个渠道：代币铸造和网络交易手续费。代币铸造按照预设的通胀曲线进行，每年铸造一定数量的新代币进入奖励池。交易手续费则按照一定比例分配进入奖励池（其余部分用于燃烧和国库）。

**基础奖励计算**

节点的数据贡献奖励采用以下计算公式：

**Node_Reward = (Node_Score / Total_Score) × Epoch_Reward × Quality_Factor**

其中，Node_Score 是节点在该 Epoch 的贡献得分，由数据采集量、数据质量、在线时长等因素综合计算得出。Total_Score 是网络中所有节点的总得分。Epoch_Reward 是该 Epoch 可分配的总奖励金额。Quality_Factor 是质量系数，根据节点的历史表现动态调整。

**Tier 权重加成**

不同 Tier 的游戏数据对应不同的权重，这直接影响节点能够获得的奖励。具体加成如下：Tier 1 游戏数据贡献权重为 1.0 倍基础得分；Tier 2 游戏数据贡献权重为 0.3 至 0.6 倍基础得分；Tier 3 游戏数据贡献权重为 0.05 至 0.15 倍基础得分。

这意味着，采集 Tier 1 游戏的节点每单位数据能够获得的奖励是 Tier 3 节点的 6 至 20 倍。这种差异化的激励设计引导节点优先采集高可信度的数据，提升整体网络的数据质量。

**探索 Bonus 机制**

为了鼓励节点拓展新的游戏数据源，PoLE 设立了探索 Bonus 机制。当节点首次成功采集并验证某个新游戏的数据（该游戏此前没有节点覆盖）时，节点将获得一次性的探索奖励。探索奖励的金额根据该游戏的潜在价值（预估 GVS）计算，通常为该游戏月均贡献奖励的 1 至 3 倍。

探索 Bonus 的设计解决了冷启动问题。在网络初期，覆盖的游戏数量较少，通过探索奖励可以快速扩大网络的覆盖范围。随着网络成熟，探索奖励会逐步降低，最终完全取消，因为届时扩展新游戏的边际收益将趋于零。

**验证节点奖励**

除了数据采集奖励外，验证节点还参与区块生产奖励的分配。区块生产奖励按照质押比例进行分配，但设置了单节点上限，防止过度集中。验证节点还需要承担数据验证的职责，其验证工作的质量将影响其获得的奖励比例。

**年化收益率模拟**

在不同网络规模下，节点的年化收益率（APR）会有所不同。以下是几种典型场景的模拟结果：

场景一（网络初期，1000 个节点）：假设网络总 GVS 为 1,000,000，单位 GVS 对应年化奖励 200 $POLE，节点平均覆盖 10 款游戏。节点年化收益 = 1,000,000 ÷ 1000 × 200 ÷ 10 = 2,000 $POLE。假设节点运营成本为 500 $POLE/年（电费 + 运维），净收益为 1,500 $POLE/年。如果初始投入（购买设备 + 质押代币）约为 15,000 $POLE，则 APR 约为 10%。

场景二（网络成长期，10,000 个节点）：假设网络总 GVS 为 50,000,000，单位 GVS 对应年化奖励 40 $POLE，节点平均覆盖 20 款游戏。节点年化收益 = 50,000,000 ÷ 10,000 × 40 ÷ 20 = 10,000 $POLE。假设运营成本为 1,000 $POLE/年，净收益为 9,000 $POLE/年。假设初始投入为 20,000 $POLE，APR 约为 45%。

场景三（网络成熟期，100,000 个节点）：假设网络总 GVS 为 1,000,000,000，单位 GVS 对应年化奖励 5 $POLE，节点平均覆盖 50 款游戏。节点年化收益 = 1,000,000,000 ÷ 100,000 × 5 ÷ 50 = 10,000 $POLE。假设运营成本为 2,000 $POLE/年，净收益为 8,000 $POLE/年。假设初始投入为 30,000 $POLE，APR 约为 27%。

从模拟结果可以看出，节点的 APR 在网络发展过程中呈现先升后降的趋势，在成长期达到峰值，然后随着网络成熟和竞争加剧逐步回归到合理水平。这种设计确保了早期参与者能够获得较高的风险回报，同时也为后期参与者保留了可持续的收益空间。

## 4.5 动态手续费机制

PoLE 采用动态手续费机制，根据网络拥堵状况自动调整 Gas 价格。这种设计既保证了网络在高峰期能够正常运作，又确保用户在低峰期能够享受低成本交易。

**技术实现**

费用模块（x/fee）实现了完整的动态手续费逻辑：

```go
type DynamicFeeParams struct {
    BaseGasPrice       sdk.Dec  // 基础 Gas 价格（默认 0.0001 pole）
    MaxGasPrice        sdk.Dec  // 最大 Gas 价格（默认 1 pole）
    GasAdjustment      sdk.Dec  // Gas 调整系数（默认 1.15x）
    CongestionThreshold sdk.Dec // 拥堵阈值（默认 50%）
    Enabled            bool     // 是否启用动态费用
}

// 获取当前 Gas 价格
func (k Keeper) GetCurrentGasPrice(ctx sdk.Context) sdk.Dec {
    params := k.GetDynamicFeeParams(ctx)
    if !params.Enabled {
        return params.BaseGasPrice
    }
    metrics := k.GetNetworkMetrics(ctx)
    congestionMultiplier := metrics.CongestionLevel.FeeMultiplier()
    currentPrice := params.BaseGasPrice.Mul(congestionMultiplier)
    // 应用最小/最大 bounds
    if currentPrice.LT(params.BaseGasPrice) {
        currentPrice = params.BaseGasPrice
    }
    if currentPrice.GT(params.MaxGasPrice) {
        currentPrice = params.MaxGasPrice
    }
    return currentPrice
}
```

## 4.6 手续费模型

手续费是 PoLE 网络经济循环的重要组成部分，它既为网络运营提供资金支持，又起到防止 Spam 攻击的作用。

**手续费结构**

PoLE 网络的手续费分为两种类型：交易手续费和数据上报手续费。交易手续费是用户进行代币转账、智能合约调用等操作时支付的费用。数据上报手续费是节点提交采集数据时支付的费用。

手续费的计算采用 Gas 机制，与以太坊类似。每种操作都有对应的 Gas 消耗量，Gas 价格由市场决定。用户可以设置更高的 Gas 价格来优先处理自己的交易，节点则倾向于优先处理高 Gas 价格的交易。

**手续费分配**

手续费收入的分配遵循以下比例：50% 进入奖励池，用于节点激励；25% 用于代币燃烧；25% 进入 DAO 国库，用于生态发展和治理活动。

这种分配比例的设计确保了手续费既能够激励节点参与网络运营，又能够通过燃烧机制创造价值捕获，同时还有足够的资金支持生态发展。

**防 Spam 机制**

为了防止网络被垃圾交易淹没，PoLE 采用了多重防 Spam 机制。首先，所有交易都需要支付基本的手续费，这提高了攻击成本。其次，对于高频小额的交易，设置了最低手续费阈值。第三，对于明显异常的流量模式，网络会启动临时限流措施。最后，恶意行为的节点将被 slash 其质押作为惩罚。

**典型手续费估算**

以下是几种常见操作的 Gas 消耗和手续费估算（假设 Gas 价格为 1 Gwei，$POLE 价格为 0.1 美元）：

基本转账：Gas 消耗 21,000，手续费约 0.0021 $POLE（约 0.00021 美元）
智能合约部署：Gas 消耗 1,000,000，手续费约 0.1 $POLE（约 0.01 美元）
数据上报：Gas 消耗 50,000，手续费约 0.005 $POLE（约 0.0005 美元）

可以看出，PoLE 网络的手续费水平非常低，普通用户完全能够负担。这对于促进网络的使用和采用至关重要。

## 4.7 代币用途

$POLE 代币在 PoLE 生态系统中承担着多种功能，这些功能相互关联、相互强化，共同构成了一个完整的价值循环。

**质押功能**

$POLE 代币最主要的用途是质押。验证节点需要质押一定数量的代币才能参与区块生产和网络治理。质押的代币作为保证金，确保节点有动力遵守网络规则。质押数量越多，节点被选中生产区块的概率越高，但存在单节点质押上限，防止过度集中。

**Gas 费用**

所有在 PoLE 网络上进行的操作都需要支付 Gas 费用，包括代币转账、智能合约执行、数据上报等。Gas 费用是网络运营的成本来源，同时也为代币创造了持续的需求。

**治理投票**

$POLE 代币持有者有权参与网络治理。治理内容包括协议参数调整（如通胀率、燃烧比例）、国库资金使用、生态项目资助等。治理投票采用代币权重投票方式，持有的代币越多，投票权重越大。

**游戏内激励**

除了上述核心功能外，$POLE 还将在游戏生态中发挥更大作用。未来，PoLE 将与游戏开发商合作，推出基于 $POLE 的游戏内激励系统。玩家可以使用 $POLE 购买游戏内物品、解锁内容、参与活动等。这种应用场景的扩展将进一步提升 $POLE 的实用价值。

**Staking 收益**

代币持有者可以选择将代币委托给验证节点，获得 staking 收益。委托不需要运行自己的节点，但能够获得与委托金额成正比的区块奖励分成。Staking 收益年化率根据网络状况浮动，通常在 3% 至 8% 之间。

PoLE 引入了经济可持续性评分（0-100分），综合评估网络健康状况：

| 指标 | 权重 | 评分标准 |
|------|------|----------|
| 节点增长率 | 20% | >10%/月 = 100分 |
| 手续费收入 | 25% | 覆盖50%成本 = 100分 |
| 通胀控制 | 20% | <2% = 100分 |
| 燃烧效率 | 20% | 燃烧>通胀 = 100分 |
| 参与度 | 15% | >50%流通 = 100分 |

正常场景下，PoLE 的可持续性评分稳定在 75-85 分，表明经济模型设计合理。

**敏感性分析**

以下是关键参数变化对网络的影响：

通胀率变化影响：
- 通胀率+5% → APR上升3-5% → 节点增长加速
- 通胀率-5% → APR下降3-5% → 可能导致节点流失

燃烧比例变化影响：
- 燃烧比例+10% → 净通胀下降约0.5%

- 交易量减半 → 燃烧收入减半 → 需提高通胀弥补

## 4.9 经济模型风险分析

任何经济模型都存在潜在的风险点，PoLE 也不例外。充分认识这些风险并制定应对策略，是确保项目长期可持续发展的关键。

**风险一：死亡螺旋**

死亡螺旋是指当代币价格下跌时，节点运营无利可图，导致节点数量减少，进而网络安全性下降，更多用户抛售代币，形成恶性循环。

应对策略：PoLE 的难度调整机制能够自动适应网络规模的变化。当节点数量减少时，单位奖励会相应提高，保持节点运营的可持续性。此外，多元化的奖励来源（代币铸造 + 手续费）确保了即使代币价格下跌，网络仍能通过手续费收入维持运营。

**风险二：巨鲸操控**

大持有量用户（巨鲸）可能通过大量质押来操控网络决策，或者通过分叉来实施双花攻击。

应对策略：PoLE 设置了单节点质押上限，限制了单一节点的影响力。同时，严格的验证节点准入机制和 Slash 惩罚提高了攻击成本，使得即使是巨鲸也难以实施有效的攻击。

**风险三：通胀失控**

如果网络使用率长期低迷，手续费收入不足以覆盖运营成本，过度依赖代币铸造可能导致通胀失控。

应对策略：PoLE 的通胀曲线是预设的并且公开透明，不存在任意增发代币的可能。同时，手续费燃烧机制在网络使用活跃时能够有效抵消铸造带来的通胀压力。

**风险四：治理失灵**

如果治理机制设计不当，可能导致决策效率低下或者少数人绑架多数人利益的情况。

应对策略：PoLE 采用了渐进式治理机制，重大决策需要较高的投票通过率，防止轻率决策。同时，治理提案有公示期，任何代币持有者都可以提出异议。

# 第五章 治理模型

## 5.1 治理架构

PoLE 采用去中心化治理模式，代币持有者通过投票的方式参与网络重大决策。治理架构分为三个层次：治理委员会、验证节点和普通代币持有者。

治理委员会由社区选举产生，任期一年。治理委员会负责处理日常治理事务，包括参数调整、预算审批、争议裁决等。治理委员会成员需要满足一定的质押门槛，并获得社区的广泛支持。

验证节点是网络运行的核心参与者，除了负责区块生产和数据验证外，还参与治理决策的投票。验证节点的投票权重与其质押金额成正比。

普通代币持有者可以通过委托投票的方式参与治理，无需运行自己的节点。委托投票允许持有者将投票权委托给自己信任的代表，由代表代为行使投票权。

## 5.2 治理流程

PoLE 的治理流程包括提案、讨论、投票、执行四个阶段。

**治理阶段演进**

PoLE 的治理将随着网络成熟度逐步演进：

- **V1 阶段（主网上线后 12 个月）**：核心团队主导治理，技术参数调整由团队决定，社区参与主要限于提案讨论。

- **V2 阶段（主网上线后 12-24 个月）**：混合治理模式，核心参数调整需社区投票通过，生态基金使用由治理委员会决定。

- **V3 阶段（主网上线 24 个月后）**：完全 DAO 治理，所有重要决策均由代币持有者投票决定。

**提案分类与阈值**

不同类型的治理提案需要不同的投票通过阈值：

| 提案类型 | 最低质押门槛 | 投票通过阈值 | 公示期 | 说明 |
|----------|--------------|--------------|--------|------|
| 轻微参数调整 | 10,000 $POLE | 简单多数（>50%） | 5 天 | 如 Gas 参数微调 |
| 中等治理事项 | 50,000 $POLE | 绝对多数（>60%） | 7 天 | 如燃烧比例调整 |
| 重大协议变更 | 100,000 $POLE | 超多数（>75%） | 14 天 | 如共识机制修改 |
| 国库资金使用 | 20,000 $POLE | 绝对多数（>60%） | 7 天 | 资金支出审批 |

**提案阶段**

任何持有一定数量 $POLE 的用户都可以发起治理提案。提案需要包括详细的背景说明、具体方案、预期影响等内容。发起提案需要锁定一定数量的代币作为保证金，如果提案未通过，保证金将被没收。

**讨论阶段**

提案发起后将进入公示期，公示期通常为 7 天。在此期间，社区成员可以在论坛、社交媒体等渠道对提案进行讨论。提案发起者需要回应社区的疑问和建议，并对提案进行必要的修改。

**投票阶段**

公示期结束后进入投票期，投票期通常为 5 天。代币持有者对自己持有的代币数量进行投票，投票选项包括支持、反对、弃权。投票结果根据代币权重计算，提案需要获得超过 50% 的支持率才能通过。

**执行阶段**

提案通过后进入执行期，技术团队将按照提案内容对网络进行升级或修改。执行期通常为 14 天，给节点运营者足够的时间来准备和更新软件。

# 第六章 安全性

## 6.1 威胁模型

PoLE 面临的安全威胁主要来自以下几个方面：

**Sybil 攻击**

Sybil 攻击是指攻击者通过控制大量虚假身份来操纵网络共识。在 PoLE 的场景中，攻击者可能试图通过运行大量节点来控制数据验证过程。应对措施包括：验证节点需要质押大量代币，提高了攻击成本；节点声誉系统会记录历史表现，异常行为会被标记。

**数据伪造**

数据伪造是指节点上报虚假的游戏数据以获取不当奖励。应对措施包括：多节点交叉验证机制确保数据真实性；中位数共识算法剔除异常值；偏差检测和 Slash 机制惩罚作弊行为。

**女巫攻击**

女巫攻击是指攻击者通过一个实体控制多个身份来获取不当优势。应对措施包括：节点身份验证机制确保每个节点对应唯一的物理实体；社交验证要求新节点获得一定数量现有节点的推荐。

## 6.2 技术安全措施

**加密通信**

所有节点间的通信都采用加密传输，使用 TLS 1.3 协议。关键数据还会进行额外签名验证，确保数据完整性和来源真实性。

**智能合约审计**

PoLE 的智能合约在部署前将经过多家权威安全审计公司的审计。审计内容包括：代码漏洞检查、权限控制验证、经济模型分析等。审计报告将公开发布，接受社区监督。

**漏洞赏金计划**

项目方将设立漏洞赏金计划，鼓励白帽黑客发现和报告潜在的安全漏洞。根据漏洞的严重程度，赏金金额从 1,000 至 100,000 $POLE 不等。

## 6.3 经济安全机制

**质押保证金**

验证节点需要质押大量代币作为保证金。如果节点出现恶意行为，其质押将被Slash。这种经济惩罚机制大大提高了攻击成本。

**渐进式惩罚**

PoLE 采用渐进式惩罚机制，首次轻微违规只会被警告和轻微罚款，但连续违规将导致越来越严厉的惩罚。这种设计给节点提供了改正的机会，同时对惯犯绝不姑息。

**紧急暂停机制**

在极端情况下（如发现重大漏洞或遭受攻击），治理委员会可以触发紧急暂停机制，暂时停止网络的部分功能。暂停后需要经过治理投票才能恢复正常运行。

# 第九章 风险与免责声明

## 9.1 技术风险

**智能合约风险**

智能合约代码中可能存在未被发现的漏洞，这些漏洞可能被攻击者利用导致资金损失。应对措施包括：进行多轮专业安全审计；建立漏洞赏金计划；采用渐进式功能发布策略。

**网络稳定性风险**

分布式网络可能面临各种技术问题，如网络延迟、节点故障等。应对措施包括：建立完善的监控系统；设计故障恢复机制；保持代码的高质量。

**数据源风险**

PoLE 依赖 Steam 等第三方平台的公开数据，如果这些平台改变数据访问策略或停止服务，将影响网络的正常运行。应对措施包括：开发多数据源备份机制；与游戏平台建立合作关系；预留技术升级空间。

**代币价格波动**

加密货币市场波动剧烈，$POLE 的价格可能发生大幅变化。投资者应该充分了解加密货币投资的风险，仅投入能够承受损失的资金。

**竞争风险**

区块链与游戏结合的领域竞争激烈，可能出现类似理念的项目。PoLE 的竞争优势在于先发优势、技术积累和社区基础。

**监管风险**

区块链行业在不同国家的监管政策存在差异，政策变化可能对项目产生影响。项目将持续关注监管动态，确保合规运营。

**经济模型失效**

虽然 PoLE 的经济模型经过了详细的设计和仿真，但在实际运行中可能出现意外情况。项目将保持经济模型的灵活性，通过治理机制进行调整。

**流动性风险**

在项目早期，代币的流动性可能不足，交易深度较浅。随着项目发展和用户增加，流动性将逐步改善。

**APR（Annual Percentage Rate）**：年化收益率。

**Gas**：燃料费用，网络操作的计算成本单位。

**Slash**：削减，指对违规节点的质押进行扣除的惩罚。
