# PoLE 零成本主网技术规格文档

**版本**：1.0

**发布日期**：2026年2月14日

**文档类型**：技术规格说明书

## 一、项目概述

### 1.1 设计目标

本文档详细描述 Proof of Live Engagement（PoLE）零成本主网的技术架构与实现方案。项目的核心目标是构建一个与比特币网络具有同等主权性的去中心化公链，同时实现用户层面的零交易成本体验。通过将传统区块链的算力消耗替换为玩家注意力贡献，PoLE 网络能够在保持去中心化安全性的前提下，为真实游戏玩家提供无需代币即可使用网络的体验。

本方案的技术愿景包含三个核心要素：首先，网络采用完全开源的技术栈，不依赖任何中心化服务或商业授权，实现真正的主权区块链特性；其次，通过 Gas Credit 机制将游戏活跃度转换为网络使用权限，使真实玩家享受零成本服务；第三，借鉴比特币网络的设计哲学，构建一个长期可持续、抗审查的分布式网络基础设施。

### 1.2 与比特币网络的对比

理解「和比特网络一样」这一需求的具体含义对于技术选型至关重要。在本项目中，这一表述包含以下具体技术特征的实现目标：

| 特性维度 | 比特币网络 | PoLE 网络 | 实现方式 |
|---------|------------|-----------|----------|
| 主权性 | 独立网络，独立共识 | 独立网络，PoLE 共识 | 自建区块链 |
| 去中心化 | 全球分布节点 | 玩家节点网络 | P2P 协议 |
| 代币经济 | 挖矿产出，有限通胀 | PoLE 挖矿，通胀可控 | 混合通胀模型 |
| 交易成本 | 需 BTC 支付 gas | 玩游戏免 gas | Gas Credit 系统 |
| 抗审查 | 开放共识 | 游戏活跃度证明 | 去中心化预言机 |

需要特别说明的是，PoLE 网络并非追求与比特币完全相同的实现方式，而是在继承其核心设计哲学的基础上，针对游戏场景进行优化。比特币采用的 PoW（工作量证明）机制虽然安全性极高，但每秒仅能处理约 7 笔交易，无法满足游戏场景对实时性的要求。因此，PoLE 采用经过改良的 PoLE+DPoS 混合共识机制，在保持去中心化特性的同时实现更高的吞吐量。

## 二、技术架构总览

### 2.1 系统分层设计

PoLE 零成本主网采用经典的分层架构设计，从底层到顶层依次包括网络层、共识层、执行层和应用层。每一层都承担特定的功能职责，层与层之间通过明确定义的接口进行交互。这种分层设计不仅提高了代码的可维护性，也为未来的功能迭代和性能优化提供了灵活性。

网络层负责节点之间的通信与数据同步，采用 Libp2p 协议实现点对点连接。Libp2p 提供了成熟的节点发现机制、消息广播功能和连接管理能力，能够支持大规模的去中心化网络。在网络层之上，共识层实现了 PoLE 核心机制，负责验证交易、确认区块和维护网络一致性。执行层处理交易的实际执行和智能合约的运行，采用 EVM（以太坊虚拟机）兼容设计以支持现有的以太坊开发者工具和智能合约。应用层则包括钱包、区块浏览器和各类 DApp，为终端用户提供交互界面。

### 2.2 技术栈选择

基于上述架构设计，PoLE 主网选择以下核心技术栈：

**底层框架选用 Cosmos SDK**。Cosmos SDK 是目前最成熟的区块链开发框架之一，提供了模块化的区块链开发架构，允许开发者通过组合不同模块来构建自定义的区块链应用。选择 Cosmos SDK 的主要考量包括：完全开源免费，无商业授权费用；支持自定义共识机制，能够完美实现 PoLE 机制；内置的 IBC（跨链通信）协议为未来的多链互联奠定基础；成熟的开发者社区和完善的文档支持。

**智能合约层采用 EVM 兼容设计**。通过集成 Ethermint 模块，PoLE 网络能够直接运行以太坊生态的智能合约。这意味着现有的 Solidity 开发者可以无缝将他们的 DApp 迁移到 PoLE 网络，同时也可以使用 MetaMask、Truffle 等成熟的以太坊开发工具。这种兼容性大大降低了生态开发的门槛。

**节点客户端使用 Rust 开发**。Rust 语言以其内存安全性和高性能著称，非常适合区块链节点的开发要求。PoLE 节点的轻量级客户端将使用 Rust 编写，确保玩家可以在家用电脑上轻松运行节点软件。

### 2.3 模块化架构设计

PoLE 采用高度模块化的架构设计，将网络功能划分为四个独立可插拔的层。这种设计不仅提高了系统的可维护性和可升级性，还允许不同模块独立演进，优化整体性能。模块化架构的核心优势在于：每个模块可以独立开发、测试和部署，大幅降低系统迭代的风险；同时，不同的实现方案可以共存，便于未来技术升级。

#### 2.3.1 数据采集层（Sensing Layer）

数据采集层是 PoLE 网络的数据入口，负责从多种数据源采集游戏活跃度数据。该层采用插件化设计，支持灵活扩展新的数据源。

**平台 API 集成模块**

- **Steam 集成**：通过 Steam Web API 获取游戏的实时在线玩家数、峰值玩家数、历史趋势等数据。支持批量查询和轮询机制，确保数据的及时性。模块内置数据缓存，减少 API 调用频率，符合平台的使用限制。
- **Epic Games Store 集成**：Epic 平台的开放 API 相对有限，模块通过页面爬取和数据聚合服务获取游戏活跃度数据。所有数据均经过交叉验证，确保可靠性。
- **第三方统计平台集成**：与 PlayerIGN、SteamDB 等第三方数据平台建立数据合作，获取更丰富的数据维度。包括但不限于：游戏评论数、关注数、销量估算等。
- **社区验证输入**：对于没有公开 API 的游戏，引入社区验证机制。节点可以提交游戏活跃度的目击证明，经过多节点交叉验证后采纳。

**数据格式标准化模块**

所有采集到的原始数据经过标准化处理后，进入统一的内部格式：

```protobuf
message GameActivityData {
    string game_id = 1;           // 游戏唯一标识
    Platform platform = 2;        // 数据来源平台
    uint32 current_players = 3;   // 当前在线人数
    uint32 peak_players = 4;      // 24小时峰值
    uint64 timestamp = 5;          // 数据时间戳
    DataSource source = 6;         // 详细数据来源
    bytes signature = 7;           // 数据签名
}
```

#### 2.3.2 数据可用性层（Data Availability Layer）

数据可用性层负责存储和管理采集到的原始数据，确保数据的持久性和可查询性。

**分布式数据存储节点**

- 采用纠删码（Erasure Coding）技术，将数据分散存储在多个节点上
- 数据冗余度可配置，默认设置为 4+2（4 个数据分片 + 2 个校验分片）
- 支持数据的快速检索和历史回溯

**数据索引服务**

- 构建高效的数据索引，支持按时间、游戏、平台等多维度查询
- 实时更新索引，确保新数据可立即被查询
- 提供数据订阅接口，支持链上合约主动拉取数据

**历史数据归档**

- 超过 90 天的数据进行归档处理
- 归档数据存储在冷存储中，降低存储成本
- 归档数据可通过治理提案解锁恢复

#### 2.3.3 计算验证层（Execution Layer）

计算验证层是 PoLE 网络的核心处理引擎，负责数据处理、验证和 GVS 计算。

**AI 驱动的异常检测引擎**

- 采用轻量级机器学习模型进行实时异常检测
- 模型运行在每个验证节点上，独立进行推理
- 检测结果通过零知识证明验证，确保计算正确性

**GVS 计算核心模块**

- 实现标准化的 GVS 计算公式
- 支持可配置的参数调整，通过治理投票修改权重
- 计算结果可审计、可追溯

**多节点交叉验证模块**

- 当单个节点的数据被质疑时，启动多节点交叉验证
- 采用 BFT 共识的子集进行快速验证
- 验证结果写入链上，作为后续争议处理的依据

**ZK 证明生成模块**

- 为计算密集型操作生成零知识证明
- 证明验证在链上进行，确保计算完整性
- 压缩链上验证成本，提升网络吞吐量

```python
计算验证层工作流程：
1. 接收数据采集层的原始数据
2. 进行格式验证和时间戳校验
3. AI 异常检测引擎初步筛选
4. GVS 核心模块计算游戏价值
5. 多节点交叉验证（如需要）
6. 生成 ZK 证明
7. 提交至结算层进行最终确认
```

#### 2.3.4 结算层（Settlement Layer）

结算层负责链上激励结算和状态确认，是 PoLE 网络与区块链交互的接口。

**区块生产和共识模块**

- 基于 Cosmos SDK 的 Tendermint 共识
- 实现 PoLE+DPoS 混合共识机制
- 支持快速finality，区块确认时间 < 3 秒

**代币分发和燃烧模块**

- 自动执行通胀产生的区块奖励分发
- 实现交易手续费的燃烧机制
- 支持惩罚性 slash 的代币处理

**治理投票模块**

- 支持链上提案和投票
- 实现代币加权的投票机制
- 自动执行通过的提案

#### 2.3.5 模块间通信协议

各层之间通过标准化的通信协议进行交互，确保系统的松耦合性。

**内部消息队列**

- 采用 gRPC 进行层间通信
- 消息格式采用 Protocol Buffers 定义
- 支持同步和异步两种调用模式

**事件驱动架构**

- 各层之间通过事件总线进行解耦
- 支持事件的持久化和重放
- 便于系统调试和审计

**服务发现机制**

- 动态服务注册和发现
- 支持服务的水平扩展
- 内置负载均衡和故障转移

## 三、共识机制详解

### 3.4 AI 融合技术

PoLE 网络积极整合人工智能技术，以提升数据处理效率和网络安全性。AI 模块作为可选增强功能，运行在计算验证层，为整个网络提供智能化的支持。

#### 3.4.1 异常检测与反作弊

AI 驱动的异常检测引擎能够识别节点上报数据中的异常模式。系统采用多层检测机制，从数据完整性、行为模式、时间序列等多个维度进行综合分析。

**第一层：数据完整性检测**

- 验证 Steam API 返回的数据格式是否符合平台规范
- 检查数据时间戳的合理性（不允许未来时间或超过 24 小时的历史数据）
- 交叉验证多个数据源的聚合数据是否一致
- 检测数据字段的完整性，确保无缺失值或异常值

**第二层：行为模式分析**

- 机器学习模型分析节点历史行为，识别可疑的数据操纵模式
- 建立节点行为画像，检测瞬间玩家数突变、规律性刷分等异常行为
- 监控节点的活跃时间分布，识别非人类操作特征
- 分析同一硬件指纹下的多开行为，检测 Sybil 攻击

**第三层：网络协同检测**

- 多节点数据交叉验证，识别串通作弊的节点集群
- 地理位置异常检测，识别使用 VPN 或代理的隐藏身份
- 网络拓扑分析，发现试图操纵共识的恶意节点网络

```python
异常检测模型输入特征：
- 节点上报频率（正常范围：5-60分钟/次）
- 数据偏差度（与全网平均值的标准差）
- 时间序列相关性（玩家数的时序平稳性）
- 硬件指纹重复率（同一硬件的节点数量）
- 历史违规记录（权重随时间衰减）
- 游戏类型匹配度（数据与游戏类型的一致性）
- 网络延迟分布（异常的高延迟可能表示数据伪造）
```

**处理流程**

当检测到异常数据时，系统自动执行以下流程：首先将疑似违规节点标记为「待审核」状态，暂停其参与共识的资格；然后将相关证据提交给治理委员会进行人工复核；最后根据复核结果做出惩罚决定。整个流程确保每个案件都得到公正处理，避免误伤正常节点。

#### 3.4.2 GVS 智能计算

传统的 GVS 计算基于固定公式和静态参数，但在快速变化的游戏市场中，这种方法可能无法准确反映游戏的真实价值。引入 AI 模型后，系统能够综合分析更多元的指标，并动态适应市场变化。

**多维度价值评估**

AI 驱动的 GVS 计算引入以下额外评估维度：

- **玩家留存率分析**：分析 1 日、7 日、30 日留存率，评估游戏的长期吸引力
- **付费转化率评估**：通过第三方数据估算游戏的付费用户比例
- **社区活跃度分析**：整合 Discord、Reddit、Twitter 等社交平台的讨论热度
- **市场趋势预测**：基于历史数据预测游戏的热度走势
- **生态互补性评估**：分析该游戏与现有 PoLE 生态的协同效应

**动态权重调整**

AI 模型能够根据市场环境自动调整各指标的权重。例如：

- 当整体市场下行时，增加「付费转化率」权重，筛选高现金流游戏
- 当某类游戏集中上线时，降低「稀缺性」权重，避免过度炒作
- 当网络规模扩大时，增加「去中心化贡献」权重，鼓励多样化参与

**预测与修正**

AI 模型不仅评估当前价值，还能预测未来价值变化：

- 预测游戏热度拐点，提前调整 GVS
- 识别潜在的爆款游戏，给予早期节点额外激励
- 检测游戏衰退趋势，触发 Tier 降级预警

```python
GVS智能计算架构：
输入层 → 特征提取 → 深度学习模型 → 权重融合 → GVS输出
         ↓
    市场信号输入
    社交媒体输入
    链上行为输入
```

#### 3.4.3 个性化节点推荐

基于 AI 的节点运营辅助系统，帮助玩家优化数据采集策略，提高参与收益。

**目标游戏推荐**

系统分析以下因素，为节点推荐高价值目标游戏：

- 该游戏的历史 GVS 稳定性和增长潜力
- 当前节点覆盖情况（优先推荐覆盖不足的游戏）
- 游戏的在线时段分布（选择玩家活跃时段采集）
- 竞争程度分析（避免过度竞争的低效采集）

**策略优化建议**

基于节点历史表现，AI 系统提供个性化运营建议：

- 最佳采集时间段推荐
- 游戏切换策略（多款游戏轮换的优化方案）
- 资源分配建议（CPU/带宽资源的最优配置）
- 风险预警（即将降级的游戏提示）

#### 3.4.4 技术实现规格

AI 模块采用轻量级设计，确保不影响网络性能：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 推理延迟 | < 10ms | 单次推理最大耗时 |
| CPU 占用 | < 5% | 节点客户端总资源占用 |
| 内存占用 | < 500MB | AI 模型和中间数据存储 |
| 模型大小 | < 100MB | 轻量化模型包 |
| 更新周期 | 每周 | 模型参数和作弊特征更新 |

**离线推理模式**

考虑到隐私和性能要求，AI 推理主要在本地节点执行：

- 模型以加密形式分发，确保算法不被泄露
- 输入数据本地处理，不上传原始玩家数据
- 仅提交推理结果和 ZK 证明，验证计算正确性

**云端协同训练**

模型的训练和更新在云端进行：

- 收集去标识化的异常模式特征
- 联邦学习框架保护节点隐私
- 定期发布模型更新，节点自动下载应用

### 3.1 PoLE 核心原理

PoLE（Proof of Live Engagement）共识机制是整个网络的核心创新。与传统区块链使用算力（PoW）或质押量（PoS）来决定出块权不同，PoLE 引入了一种全新的维度——玩家的真实游戏活跃度。这种设计从根本上解决了两个问题：一是将玩家的游戏行为转化为网络价值，实现「玩游戏即挖矿」；二是通过活跃度证明来分配出块权重，确保只有真实玩家才能参与网络共识。

PoLE 的工作原理可以分解为以下几个步骤：首先，玩家在本地运行节点软件，该软件通过操作系统的进程检测功能识别当前正在运行的游戏程序；然后，节点软件调用 Steam 公开 API 获取游戏的当前在线玩家数作为基准数据；接下来，节点将收集到的数据进行签名处理，生成活跃度证明并广播到网络；最后，网络中的验证节点对活跃度证明进行共识验证，根据验证结果确定出块权重。

这种机制的关键优势在于它利用了 Steam 公开 API 的实时数据作为可信来源。由于 Steam API 返回的是平台级别的聚合数据而非单个用户数据，因此玩家无法伪造自己的活跃度证明。同时，由于所有数据都需要多个节点的独立采集和交叉验证，网络能够有效识别并惩罚任何试图操纵数据的恶意行为。

### 3.2 混合共识架构

虽然 PoLE 机制能够有效验证玩家的游戏活跃度，但单纯依赖活跃度来确定出块权存在理论上的安全隐患——攻击者可能通过控制大量游戏账号来操纵网络。为解决这一问题，PoLE 采用了 PoLE+DPoS 混合共识架构，在保证去中心化的同时提升网络安全性。

在混合共识中，出块权重由两个因素共同决定：质押权重和活跃度权重各占 50%。质押权重的计算方式与传统的 DPoS 机制相同，即节点质押的代币数量越多，权重越大。活跃度权重则根据节点关联的游戏活跃度证明计算得出。这种双重权重机制确保了网络安全性——即使攻击者能够伪造大量游戏活跃度证明，如果没有足够的代币质押，仍然无法控制网络。

具体的出块权重计算公式如下：

```
W_total = W_stake × 50% + W_engagement × 50%

其中：
W_stake = min(StakeAmount / MaxStake, 1.0) × 100
W_engagement = min(EngagementScore / TargetScore, 1.0) × 100
```

在这个公式中，StakeAmount 是节点质押的代币数量，MaxStake 是网络中单个节点的最大允许质押量（用于防止巨鲸垄断）。EngagementScore 是节点在特定周期内积累的活跃度分数，TargetScore 则是网络设定的目标活跃度基准。

### 3.3 验证者选举与轮换

PoLE 网络的验证者集合通过质押排名和活跃度评分的综合排名来确定。每 24 小时，网络会根据以下规则重新选举验证者集合：

首先，所有满足最低质押要求的节点按照总权重（W_total）进行排名；然后，排名前 21 位的节点被选为当前周期的活跃验证者；最后，当选的验证者按照权重比例分配出块任务。这种设计确保了验证者集合的动态性和公平性，防止少数节点长期垄断出块权。

验证者的最低质押要求设定为 10,000 $POLE，这一数值足以确保验证者有足够的經濟激励维护网络正常运行，同时不会过高以至于阻碍普通玩家参与。对于活跃度权重达到网络前 100 名的节点，最低质押要求可降低至 1,000 $POLE，以此激励更多真实玩家参与网络共识。

## 四、零成本实现机制

### 4.1 Gas Credit 系统设计

实现「零成本」用户体验是 PoLE 网络的核心设计目标之一。然而，如果简单地取消所有交易费用，网络将面临严重的 DDoS 攻击风险。历史上，无数区块链项目因为忽视了这一点而导致网络瘫痪。因此，PoLE 采用了一种巧妙的解决方案——Gas Credit（燃料信用）系统。

Gas Credit 系统的核心思想是将网络使用权限与游戏活跃度挂钩。玩家通过玩游戏获得的不是直接的代币奖励，而是 Gas Credit——一种用于支付交易费用的不可转让的信用额度。当玩家想要发起交易时，系统会优先消耗其 Gas Credit 账户中的余额，只有当 Gas Credit 不足时才需要使用原生代币支付费用。

这种设计实现了多赢局面：对于真实游戏玩家而言，他们在玩游戏的过程中自动积累 Gas Credit，可以无限免费地使用网络；对于网络本身，由于 Gas Credit 与真实游戏活跃度绑定，攻击者无法大量获取信用来发起攻击；对于整个生态系统的价值流转，虽然玩家不需要直接购买代币来支付费用，但活跃的玩家行为本身为网络提供了有价值的输入数据，这些数据通过 PoLE 机制转化为网络的整体价值。

### 4.2 Gas Credit 获取规则

玩家获取 Gas Credit 的方式与他们的游戏活跃度直接相关。具体规则如下：

**基础获取规则**：每当玩家运行游戏客户端并保持游戏进程活跃时，节点软件会自动采集活跃度数据并生成证明。每完成一次有效的数据上报，玩家将根据该游戏所在 Tier 级别获得相应的 Gas Credit 奖励。Tier 1（Steam 高活跃游戏）的标准奖励为 10 信用点每次，Tier 2 为 5 信用点每次，Tier 3 为 1 信用点每次。

**时间累积规则**：为了鼓励玩家持续参与，单次游戏会话的时间越长，获得的信用奖励倍率越高。具体而言，连续游戏 1 小时以内为 1 倍率，1-3 小时为 1.5 倍率，3-6 小时为 2 倍率，6 小时以上为 3 倍率。这一机制有效防止了玩家通过频繁进出游戏来刷取信用。

**日上限规则**：每个玩家每天通过游戏活跃度最多可获得 1,000 Gas Credit（按照最高倍率计算）。这一上限既能保证大多数玩家的正常使用需求，又能防止通过多账号或脚本进行信用刷取。

### 4.3 交易费用计算

在 PoLE 网络中，每笔交易的实际费用计算遵循以下逻辑：

```
if (GasCredit >= RequiredGas) {
    // 使用 Gas Credit 支付
    Fee = 0 (对用户完全免费)
    GasCredit -= RequiredGas
} else {
    // 使用原生代币支付
    Fee = RequiredGas × GasPrice
    // GasPrice 动态调整，基础价格为 0.0001 $POLE
}
```

RequiredGas 是交易实际消耗的 Gas 数量，不同类型的交易有不同的 Gas 消耗标准。例如，普通转账交易消耗 21,000 Gas，智能合约部署根据复杂程度消耗 100,000 至 1,000,000 Gas 不等。

对于没有 Gas Credit 或 Gas Credit 不足的用户，网络会收取正常的交易费用。这确保了网络在经济上的可持续性——即使大量用户使用零成本服务，仍有部分用户（可能是开发者、机器人或恶意攻击者）需要支付费用来维护网络安全。

### 4.4 手续费分配机制

对于需要支付原生代币的交易，其手续费按照以下比例进行分配：

```
总手续费 = 50% 燃烧 + 30% 验证者奖励 + 20% 生态基金
```

燃烧机制是 PoLE 经济模型的重要组成部分。通过将部分手续费进行燃烧处理，网络实现了逐步通缩的效果，有助于代币价值的长期稳定。验证者奖励直接分配给维护网络运行的节点，确保他们有足够的经济激励继续提供服务。生态基金则用于资助生态内的 DApp 开发、流动性激励和其他有助于网络发展的项目。

## 五、代币经济模型

### 5.1 代币参数

PoLE 网络的原生代币 $POLE 具有以下基本参数：

| 参数项 | 数值 | 说明 |
|--------|------|------|
| 代币名称 | POLE | 项目主代币 |
| 代币符号 | $POLE | 交易符号 |
| 总供应量 | 10 亿枚 | 初始发行量 |
| 最小单位 | 10^-18 | 类似于 Gwei |
| 代币类型 | 原生代币 | 非 ERC-20 |
| 最大通胀率 | 20%/年 | 主网第一年 |

代币的初始分配比例经过精心设计，以确保各方利益得到合理平衡：

```
初始分配：
├── 节点奖励：60%（6 亿枚，通过通胀渐释）
├── 生态基金：20%（2 亿枚，用于 DApp 激励）
├── 社区空投：15%（1.5 亿枚，早期参与者）
└── 团队/投资者：5%（5,000 万枚，2 年锁仓）
```

### 5.2 通胀机制

PoLE 网络采用逐步递减的通胀机制，类似于比特币的减半设计，但参数经过调整以适应游戏场景的需求：

```
年度通胀公式：
InflationRate(year) = 20% × (0.5)^(floor((year-1)/2))

各年通胀率：
├── 第1年：20.0%
├── 第2年：20.0%
├── 第3年：10.0%
├── 第4年：10.0%
├── 第5年：5.0%
├── 第6年：5.0%
├── 第7-8年：2.5%
├── 第9-10年：1.25%
└── 第10年后：≤1.0%（长期均衡）
```

这种通胀设计的核心理念是：网络启动初期需要大量吸引节点参与，因此采用较高的通胀率；随着网络规模扩大和生态成熟，通胀率逐步降低以保护持币者利益；长期来看，通胀率维持在 1% 以下，既能激励网络维护者，又能避免过度通胀。

### 5.3 燃烧机制

除了通胀减半机制外，PoLE 还设计了多重燃烧机制来对抗通胀压力：

**交易手续费燃烧**：每笔交易手续费的 50% 直接被销毁。这部分燃烧量与网络活跃度直接相关——网络越活跃，燃烧量越大。在牛市中，高交易量可能导致净通缩效果。

**惩罚性燃烧**：当验证者被发现有恶意行为（如提交虚假活跃度证明、试图分叉网络）时，其质押的代币将被 slash（罚没）。被罚没代币的 50% 进行永久燃烧，剩余 50% 分配给举报者或生态基金。

**协议燃烧**：DAO 治理可以提案触发额外的代币燃烧。例如，当网络积累过多闲置代币时，可以将生态基金中的一部分代币进行燃烧以提振代币价值。这类燃烧需要 51% 以上的代币持有者投票通过。

### 5.4 奖励分配模型

验证者和节点的奖励来自两个主要来源：区块奖励和手续费分成。

**区块奖励**：每个区块铸造的代币数量根据通胀公式计算。在网络启动初期，每个区块（约 6 秒）奖励约为 50 $POLE。随着时间推移，区块奖励将按照通胀率递减。

**手续费分成**：验证者可以分享当前区块中包含的所有交易手续费。按照共识规则，当期的活跃验证者按照其权重比例分配手续费。

**活跃度额外奖励**：为了激励玩家参与，节点软件还会根据玩家贡献的活跃度数据单独发放 Gas Credit 形式的奖励。这部分奖励来自通胀产生的代币，不影响验证者的收益。

## 六、节点架构与部署

### 6.1 节点类型设计

PoLE 网络定义了三种类型的节点，每种节点承担不同的功能职责：

**验证节点（Validator Node）**：验证节点是网络的核心，负责打包区块和确认交易。成为验证节点需要质押最低数量的 $POLE（默认 10,000 枚）。验证节点需要运行稳定的服务器环境，保持 99% 以上的在线时间。验证节点的硬件建议配置为：8 核 CPU、16GB RAM、500GB SSD 存储空间。

**全节点（Full Node）**：全节点同步并存储网络的全部历史数据，为网络提供数据可用性保障。任何人都可以运行全节点，无需质押要求。全节点的硬件建议配置为：4 核 CPU、8GB RAM、1TB SSD 存储空间。存储空间需求会随网络规模增长而增加。

**轻节点（Light Node）**：轻节点是 PoLE 网络的特色创新，它集成在玩家的游戏客户端中。轻节点不存储完整的区块链数据，仅保存最新的区块头信息用于验证交易。轻节点的主要功能是采集和上报游戏活跃度数据，是 PoLE 共识的数据来源。轻节点对硬件无特殊要求，只需能够运行游戏客户端即可。

### 6.2 零成本运营策略

实现「零成本」运营是 PoLE 网络的重要目标。为达成这一目标，项目采用以下策略：

**社区节点捐赠计划**：早期网络完全依赖社区成员提供的服务器资源运行。贡献服务器作为验证节点的个人或组织将获得额外的 $POLE 代币奖励作为回报。这种模式类似于比特币早期的挖矿社区，具有高度的去中心化特性。

**玩家节点激励**：对于运行轻节点的普通玩家，虽然他们不参与区块生产，但通过 PoLE 机制贡献的活跃度数据对于网络价值至关重要。玩家通过玩游戏自动积累 Gas Credit，同时也可以获得一定数量的 $POLE 代币奖励。

**存储优化技术**：通过实现状态树剪枝和历史数据归档功能，大幅降低全节点的存储需求。普通用户运行全节点的门槛将显著降低，有助于网络的进一步去中心化。

### 6.3 节点软件架构

节点软件采用模块化设计，包含以下核心组件：

```
┌─────────────────────────────────────────────────────────────────┐
│                        节点软件主进程                             │
├─────────────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐ │
│  │ P2P 网络模块 │  │ 共识引擎    │  │ 交易池管理          │ │
│  └──────────────┘  └──────────────┘  └──────────────────────┘ │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐ │
│  │ 状态数据库   │  │ API 服务器  │  │ 轻节点桥接          │ │
│  └──────────────┘  └──────────────┘  └──────────────────────┘ │
├─────────────────────────────────────────────────────────────────┤
│                      游戏活跃度采集模块                           │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐ │
│  │ 进程检测器  │  │ Steam API   │  │ 数据签名器          │ │
│  └──────────────┘  └──────────────┘  └──────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**进程检测器**：运行在操作系统层面，实时检测游戏进程的启动和退出。检测到游戏进程后，提取游戏的唯一标识符（如 Steam appid）用于后续处理。

**Steam API 桥接**：调用 Steam 公开 API 获取游戏的当前在线玩家数据。该数据将作为活跃度证明的重要组成部分。

**数据签名器**：将采集到的活跃度数据打包成标准格式，使用节点的私钥进行签名，然后广播到网络。

## 七、安全机制

### 7.1 威胁模型分析

PoLE 网络面临的主要安全威胁包括以下几类：

**Sybil 攻击**：攻击者试图通过创建大量虚假节点来控制网络。在 PoLE 机制下，单纯创建大量节点是不够的——每个节点必须通过质押代币和贡献真实活跃度才能获得权重。这大大提高了 Sybil 攻击的成本。

**数据操纵**：攻击者试图伪造游戏活跃度数据以获取不当奖励。由于活跃度数据来源于 Steam 公开 API，攻击者无法直接伪造平台数据。但攻击者可能试图通过控制大量游戏账号来人为制造活跃度。为此，PoLE 引入多维度验证机制来检测异常模式。

**51% 攻击**：攻击者试图控制网络中大部分的验证者投票权。在 PoLE 的混合共识机制下，即使攻击者控制了 51% 的质押代币，如果无法同时控制足够的活跃度权重，仍无法完全主导网络。

### 7.2 防护措施

针对上述威胁，PoLE 实现了多层次的安全防护机制：

**质押门槛与 slash 机制**：验证节点必须质押最低数量的代币，且任何违规行为都将导致部分或全部质押代币被罚没。具体规则包括：连续 3 次未能及时签名区块 → 罚没 10% 质押；提交虚假活跃度证明 → 罚没 50% 质押并永久取消验证资格；试图分叉或双花 → 罚没全部质押并列入黑名单。

**活跃度验证**：网络会对节点提交的活跃度证明进行多维度验证。验证内容包括：数据来源的合法性（仅接受 Steam 等认证平台的数据）、时间戳的合理性（不能是未来时间或过旧的时间）、数据的一致性（多个节点对同一游戏的数据偏差应在合理范围内）。任何明显异常的证明都将被拒绝，提交者将受到惩罚。

**冷却期机制**：新质押的代币需要等待 7 天才能参与共识投票。这一机制防止攻击者通过快速质押大量代币来发动闪电攻击。

### 7.3 审计与漏洞赏金

主网上线前，PoLE 项目将进行全面的安全审计，审计范围包括：共识机制的安全性、智能合约的漏洞检查、网络协议的安全性分析。审计将由至少两家独立的第三方安全公司执行。

同时，项目将设立漏洞赏金计划，鼓励白帽子黑客发现并报告潜在的安全漏洞。发现的漏洞将按照严重程度分级给予奖励，严重漏洞的最高奖励可达 100,000 $POLE。

## 八、路线图与实施计划

### 8.1 阶段一：基础框架搭建（第 1-4 周）

本阶段的主要目标是搭建 PoLE 网络的基础技术框架。具体工作包括：完成 Cosmos SDK 基础链的配置和定制；实现 PoLE 共识机制的核心算法；搭建 P2P 网络通信层；开发基本的节点客户端程序。阶段结束时，网络将具备基本的区块生产和交易处理能力。

### 8.2 阶段二：核心功能实现（第 5-10 周）

本阶段实现网络的核心业务逻辑。主要工作包括：开发 Gas Credit 系统的完整逻辑；集成 Steam API 预言机模块；实现 Tier 分层和 GVS 计算引擎；开发区块浏览器和钱包的基础功能。阶段结束时，网络将能够支持真实玩家参与并获得奖励。

### 8.3 阶段三：测试与优化（第 11-14 周）

本阶段进行全面的系统测试和性能优化。主要工作包括：内部测试网的多轮压力测试；安全审计和漏洞修复；性能调优（目标 TPS 达到 1,000 以上）；文档完善和开发者工具链准备。

### 8.4 阶段四：主网上线（第 15-16 周）

本阶段完成主网的正式上线。主要工作包括：主网启动脚本和配置文件准备；创世区块的生成和分发；初始验证者集合的启动；监控和应急响应机制就位。

## 九、总结

本文档详细描述了 PoLE 零成本主网的技术架构和实现方案。通过采用 Cosmos SDK 作为底层框架、结合 PoLE+DPoS 混合共识机制、引入 Gas Credit 零成本系统，PoLE 网络能够在实现「和比特网络一样」的主权性和去中心化特性的同时，为游戏玩家提供零成本的优质体验。
